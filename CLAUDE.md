# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository layout

npm workspaces monorepo with three packages that build in order:

1. `schemas/` — published as `mwpa_schemas`. Runtime types/validators (using `vts`) consumed by both backend and frontend. **Do not hand-edit files under `schemas/src/`** — they are generated by `vtseditor` from `schemas/schemas.json` (see `vtseditor.json`). Edit the JSON via the vtseditor UI or its MCP tools (`mcp__vtseditor__vts_*`), then regenerate.
2. `backend/` — Node.js 18+ ES module (`NodeNext`) Express app. Built on the `figtree` framework (`BackendApp`, `HttpService`, `MariaDBService`, `DBRepository`, `DefaultRoute`).
3. `frontend/` — Browser SPA bundled with webpack via gulp. Uses AdminLTE 3 + jQuery + Bootstrap 4 + the in-house `bambooo` widget library (vendored as a git submodule under `frontend/node_modules/bambooo`).

`old-backend/` holds the pre-refactor codebase (legacy `app/` controllers and `inc/` infrastructure). New work lives in `backend/src/`. Do not modify `old-backend/` unless porting code from it.

`package.json` declares `packageManager: "yarn@..."` for historical reasons but this repo uses **npm** for installs, updates, and workspace commands.

## Common commands

From repo root:

```bash
# install all workspaces
npm install

# build everything: schemas → backend → frontend (gulp default = copy-data + webpack)
npm run compile

# bump deps interactively across all three packages
npm run npm-check-updates
```

Per-package (use `-w <pkg>` from the repo root):

```bash
# backend type-check + emit to backend/dist
npm run compile -w backend

# schemas type-check + emit to schemas/dist
npm run compile -w schemas

# frontend asset copy (admin-lte plugins, fonts, bambooo css)
npm run gulp-copy-data -w mwpa_frontend

# frontend webpack bundle (writes to frontend/dist/{index,login,service-worker}.js)
npm run gulp-build-webpack -w mwpa_frontend
```

Local dev infra: `docker compose up -d mariadb` brings up MariaDB on `127.0.0.1:3306` (root/test, db `mwpa`, see `.env`). The backend itself runs locally, not in Compose. Start it with `node backend/dist/main.js --config=config.dev.json` after `npm run compile`. The HTTP server binds to `https://localhost:3000` with a self-signed in-memory cert — use `curl -k` or accept the cert in the browser.

The figtree logger tries to write to `/var/log/mwpa/`; on a dev machine without that directory writable, it logs `EACCES` once and falls back to console — harmless.

There is **no test suite** in the repo at this time.

## Architectural patterns

### Backend bootstrap (figtree framework)

`backend/src/main.ts` instantiates `Backend` (extends `figtree`'s `BackendApp`). The framework handles config loading (`MWPAConfig`), arg parsing (`SchemaDefaultArgs`), and a `_serviceManager` that runs `MariaDBService` (loads TypeORM entities via `MWPADbLoader`) and `HttpService` (loads routes via `RouteLoader`). ACL is registered globally with `ACL.getInstance().addController(new ACLRbac())`.

Entity registration: add new entities to `backend/src/Db/MariaDb/MWPADbLoader.ts::loadEntities()`. Route registration: add new top-level route classes to `backend/src/Routes/RouteLoader.ts::loadRoutes()`.

### Route pattern

Each top-level resource has a `Routes/Main/<Resource>.ts` class extending `DefaultRoute` that wires endpoints in `getExpressRouter()` via `this._get(path, requiresAuth, handler, {bodySchema, responseBodySchema, sessionSchema, sessionInit, description})`. The actual handler logic is split into per-endpoint files under `Routes/Main/<Resource>/<Action>.ts` (e.g. `Group.ts` → `Group/List.ts`, `Group/Save.ts`). Schemas come from `mwpa_schemas`; the framework validates request/response automatically and feeds the OpenAPI/Swagger UI route.

Mobile routes live in `Routes/Mobile/*` and respond with `{statusCode: number}` (numeric, MobileV1 envelope), while `/json/*` responses keep figtree's string-enum statusCode (e.g. `"200"`). The helper `Routes/Mobile/MobileV1.ts` exposes `MobileV1StatusCode` constants and a `toV1(res)` adapter for reusing Main handlers under `/mobile/*` (see `Routes/Mobile/Reuse.ts`).

When adding endpoints: define request/response shapes in `schemas/schemas.json` first (via `mcp__vtseditor__vts_*`), let the auto-generator update `schemas/src/`, export from `schemas/src/index.ts`, then import in the route. Do not bypass the schema validation by typing `data.body` loosely.

### Auth and session helpers (figtree gotcha)

figtree's built-in `DefaultRouteCheckUserIsLogin` validates the request against the *base* `SchemaRequestData` (with `userid: string`), and `Session.defaultInitSession()` returns `{isLogin:false, userid:'', role:''}` — both of which collide with our extended `SchemaMWPASessionData` (`userid: number`, plus `isAdmin/isMobileLogin/main_group_id/main_organization_id/groups/organizations`). Using the figtree defaults blocks every authenticated request with a 401 or session-validation 500.

For new routes, always use the MWPA-specific helpers:

- `checkMWPAUserIsLogin` (from `Routes/AuthCheck.ts`) — pass instead of `true` as the `requiresAuth` argument to `_get/_post`.
- `isMWPAUserLogin(req)` — boolean check inside handler bodies (e.g. `/json/islogin`, `/mobile/islogin`).
- `defaultMWPASessionInit` (from `Routes/SessionDefault.ts`) — set as `sessionInit` in any route description that has `sessionSchema: SchemaMWPASessionData`.

### Multipart uploads

For multipart endpoints, attach a `parser` in the route description (e.g. `multer({storage: multer.memoryStorage()}).single('file')`) — figtree's `DefaultRoute._post(...)` mounts it as middleware so `req.file.buffer` is populated. See `Routes/Mobile/Sightings.ts` `/mobile/sighting/image/save` for the canonical example.

### Repositories

Repositories extend `DBRepository<Entity>` (figtree) and expose a `getInstance()` returning `super.getSingleInstance(Entity)`. `REGISTER_NAME` must be set. Custom queries go on the subclass; default CRUD comes from the base class. Add new repositories alongside `Db/MariaDb/Repositories/`.

### ACL / RBAC

`ACLRbac.ts` defines `ROLES`, `RIGHTS`, and `ASSOCIATIONS` consumed by `rbac-simple`. Roles (`Role.root`, `Role.user`) and rights (`MwpaRights`, `RightService`, etc.) are sourced from `mwpa_schemas` — extend them in `schemas/schemas.json` rather than hardcoding strings.

### ESM import paths

Backend and schemas are ESM (`"module": "NodeNext"`). All relative imports must include the `.js` extension (e.g. `import {List} from './Group/List.js';`) even though the source is `.ts`. The frontend (webpack) uses extensionless TS imports — don't mix the two conventions.

### Frontend page/API split

`frontend/src/inc/Api/<Resource>.ts` wraps the JSON endpoints (uses `NetFetch`, throws `UnauthorizedError` on 401). `frontend/src/inc/Pages/<Resource>.ts` (+ `Pages/<Resource>/...Modal.ts`) renders the page and modals using `bambooo`. Webpack entries are `index.ts`, `login.ts`, `service-worker.ts`.

## Active refactor

The repo is mid-migration from `old-backend/` (legacy `routing-controllers`-based express controllers, custom session management, hand-written DB setup) to `backend/src/` (figtree-based). Most Main and Mobile resources are ported. When extending a feature, first check `old-backend/app/Main|Mobile/<Resource>.ts` for any logic the new side hasn't picked up yet, then port what's needed — don't reintroduce the old `routing-controllers` decorators or `MariaDbHelper.getConnection().getRepository(...)` patterns into the new tree.

## Database backwards compatibility

Production DB has live data. The TypeORM entities in `backend/src/Db/MariaDb/Entities/` are schema-equivalent to the legacy ones (same table names, columns, types, defaults, indices) — confirmed by diffing across the rename commit. figtree's `MariaDBService` runs with the default `{migrationsRun: true, synchronize: true}`, which is currently safe because of that equivalence.

If you change an entity in a way that *removes* or *narrows* columns, do **not** rely on `synchronize: true` — switch to explicit migrations via `MWPADbLoader.loadMigrations()` first, otherwise the next boot will drop production data. Adding nullable columns or new indices is fine.

## Schemas workflow

`schemas/schemas.json` is the source of truth (vtseditor format). Regeneration writes to `schemas/src/` with `destinationClear: true` — uncommitted manual edits there will be lost. The generator emits both runtime validators (`SchemaXxx`) and TS types (`Xxx`); both are re-exported from `schemas/src/index.ts`. After regenerating, run the schemas + backend compile to surface type errors in callers.
